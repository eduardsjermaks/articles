## 5. Critical flows

### 5.1 Order placement (checkout → payment → persistence → side-effects)
- **Trigger/entrypoint**: `CheckoutController` (public storefront) collects addresses/shipping/payment and calls `IOrderProcessingService` (src/Presentation/Nop.Web/Controllers/CheckoutController.cs:33-60 — the controller injects `IOrderProcessingService`, `IPaymentService`, `IShoppingCartService`).
- **Steps** (FACT — src/Libraries/Nop.Services/Orders/OrderProcessingService.cs:1594-1720):
  1. `PlaceOrderAsync` validates the request and builds a `PlaceOrderContainer` via `PreparePlaceOrderDetailsAsync` (:1594-1603).
  2. Payment is processed (`GetProcessPaymentResultAsync`, :1610-1612) — delegates to the configured payment plugin (**INFERENCE** from `IPaymentPluginManager` usage in CheckoutController.cs:51).
  3. On success: order saved (`SaveOrderDetailsAsync`), cart items moved to order items, discount & gift-card usage history saved, recurring payment created if applicable (:1616-1632).
  4. Side-effects: notifications sent, checkout data reset, activity logged, `OrderPlacedEvent` published, order status checked, `ProcessOrderPaidAsync` if already paid (:1633-1651).
- **Failure modes** (FACT): exceptions are caught, logged via `ILogger`, and surfaced as result errors — the order simply fails (:1661-1676). Duplicate-submission protection uses a **named OS `Mutex` keyed by customer id** plus a cache flag, with explicitly synchronous `.Result`/`.Wait()` calls because "mutexes cannot be used with the await operation" (:1680-1718). **INFERENCE**: this is single-machine protection only; in a web farm the mutex does not span nodes (the cache flag partially does, if the cache is distributed). This is the costliest place for errors: payment succeeded but order-save failed would need manual reconciliation — **Tentative**, compensation logic was not located.

### 5.2 Scheduled/background tasks
- **Trigger**: in-process `TaskScheduler` timers (src/Libraries/Nop.Services/ScheduleTasks/TaskScheduler.cs:263-267) POST to `{storeUrl}/scheduletask/runtask` (:67, 195-206).
- **Steps**: `ScheduleTaskController.RunTask(taskType)` resolves and runs the task via `IScheduleTaskRunner` (src/Presentation/Nop.Web/Controllers/ScheduleTaskController.cs:23-…). Example tasks: `ClearLogTask` (Logging/), `DeleteInactiveCustomersTask` (Gdpr/).
- **Failure modes** (FACT): HTTP errors are caught and logged with the task type and URL (TaskScheduler.cs:221). **INFERENCE**: if the site is idle/stopped, tasks don't run — hence KeepAlive.

### 5.3 Application startup / plugin loading
- **Trigger**: `Program.Main` → `ConfigureApplicationServices` → `NopEngine`, which discovers `INopStartup` and `IStartupTask` implementations by reflection across all assemblies including plugins (src/Presentation/Nop.Web/Program.cs:11-50; src/Libraries/Nop.Core/Infrastructure/NopEngine.cs:40-150).
- **Side effects**: DB migrations can run at startup via `MigrationManager`/`NopDbStartup` (src/Libraries/Nop.Data/Migrations/MigrationManager.cs, src/Libraries/Nop.Data/NopDbStartup.cs). **Tentative** on exact conditions (fresh install vs upgrade) — inspect `NopDbStartup.cs` and `Installation/` migrations.
- **Failure modes**: **INFERENCE** — a bad plugin DLL in `Plugins/` participates in reflection scanning and DI registration and can take down startup; the `ClearPluginAssemblies` build step (src/Build/) exists to manage plugin assembly conflicts.

## 10. Migration hotspots / tech debt (evidence-based)

1. **`OrderProcessingService` god class + hidden side-effects**
   - Evidence: 3,561 lines in one file (src/Libraries/Nop.Services/Orders/OrderProcessingService.cs); `PlaceOrderAsync` orchestrates payment, persistence, inventory movement, notifications, events, and recurring payments inline (:1594-1651).
   - Risk type: Mixed concerns | Hidden side-effects. Confidence: **High**.
   - Refactoring seam: it already publishes `OrderPlacedEvent` — extract post-order side-effects (notifications, activity log, discount/gift-card history) into event consumers behind the existing `IEventPublisher`.
   - Facade: keep `IOrderProcessingService.PlaceOrderAsync` as the stable facade; introduce `IOrderPlacementPipeline` steps internally. Test strategy: characterization tests over `PlaceOrderAsync` using the existing SQLite test harness (BaseNopTest) before splitting. Migration risk: **High** (money path).

2. **Sync-over-async + process-local mutex in order placement**
   - Evidence: `.Result` / `.Wait()` inside the mutex block and an OS-named `Mutex` for dedup (OrderProcessingService.cs:1680-1718, with an in-code comment explaining the constraint).
   - Risk type: Hidden side-effects | Coupling (to single-node hosting). Confidence: **High** (2 refs: code + README's web-farm claim at README.md:17 implies multi-node deployments where this doesn't protect).
   - Seam: replace with a distributed lock keyed through `IStaticCacheManager` (already used for the interval flag). Facade: none needed. Tests: concurrency test with two parallel `PlaceOrderAsync` calls. Risk: **Med**.

3. **Service-locator pattern (`EngineContext.Current` / `Singleton<T>`)**
   - Evidence: 137 usages of `EngineContext.Current` across src (grep count); `Singleton<AppSettings>.Instance` read directly in Program.cs:25; EngineContext at src/Libraries/Nop.Core/Infrastructure/EngineContext.cs:19-46.
   - Risk type: Coupling | Shared mutable state. Confidence: **High**.
   - Seam: constructor injection is already the norm in services (see CheckoutController's DI fields) — burn down `EngineContext.Current` call sites module by module. Risk: **Med** (mechanical but wide).

4. **Background jobs via self-HTTP inside the web process**
   - Evidence: TaskScheduler posts to `scheduletask/runtask` on the store's own public URL (TaskScheduler.cs:67,206; NopTaskDefaults.cs:11; ScheduleTaskController.cs:23).
   - Risk type: Configuration risk | Hidden side-effects (depends on store URL being correct/reachable from itself; jobs silently stop when app idles). Confidence: **High**.
   - Seam/extraction candidate: `IScheduleTaskRunner` is already an interface — extract a hosted-service (`IHostedService`) or external worker that calls the runner directly, bypassing HTTP. Facade: keep `IScheduleTaskRunner`. Tests: task-runner unit tests exist scaffolding-wise (Services test folder). Risk: **Low–Med**.

5. **Runtime-writable configuration & plugin directory in the app folder**
   - Evidence: app writes `App_Data/appsettings.json` (AppSettingsHelper.cs:28; DataSettingsManager.cs:111-119); plugins build straight into `Nop.Web/Plugins` (Payments.Manual.csproj:12-13); Dockerfile must chmod app dirs writable (Dockerfile:20-33).
   - Risk type: Configuration risk | Coupling (immutable-infrastructure unfriendly; complicates web farms and read-only file systems). Confidence: **High**.
   - Seam: config already flows through `IConfiguration` + env vars (Program.cs:20) — move secrets to env/vault and treat file-write paths as install-time only. Risk: **Med**.

6. **Reflection-based plugin discovery affecting startup**
   - Evidence: `NopEngine` scans all types for `INopStartup`/`IStartupTask` including plugin assemblies (NopEngine.cs:40-150); custom `ClearPluginAssemblies` build step exists to manage plugin DLL conflicts (src/Build/).
   - Risk type: Coupling | Hidden side-effects. Confidence: **Medium** (behavioral blast radius is inferred, not demonstrated).
   - Seam: plugin surface is already interface-driven (`IPluginService`, plugin base classes in Nop.Services/Plugins/) — candidates for out-of-process extraction are the plugin categories with clean interfaces: **payments, shipping, tax** providers. Risk: **High** (ecosystem compatibility).

7. **Thin test coverage relative to code size**
   - Evidence: 153 test files vs ~2,548 production files (find counts); CI runs tests but no coverage threshold (.github/workflows/dotnet.yml).
   - Risk type: Lack of tests. Confidence: **High** for the ratio; **Low** for any specific untested-module claim without per-module inspection.

## 5. Critical flows

### Flow A — Place an order (checkout → persistence → side-effects)
- **Trigger/entrypoint** — **FACT**: `OrderProcessingService.PlaceOrderAsync(ProcessPaymentRequest)` (`src/Libraries/Nop.Services/Orders/OrderProcessingService.cs:1594`). Called from checkout controllers/factories (public checkout). **Tentative** on exact caller — inspect `Controllers/CheckoutController`.
- **Steps (with code references)** — **FACT**:
  1. Validate request + order GUID (`:1596-1599`).
  2. Prepare & validate cart, checkout attributes, addresses, prices (`PreparePlaceOrderDetailsAsync`, `:1602`; validation at `:489-502`).
  3. Process payment via the selected provider (`_paymentService.ProcessPaymentAsync` / recurring variants, `:1610-1614`, `:1414-1424`).
  4. On success: `SaveOrderDetailsAsync` (`:1616`), move cart items → order items (`:1621`, event `ShoppingCartItemMovedToOrderItemEvent` `:1343`), record discount + gift-card usage (`:1624-1627`).
  5. Notifications & order notes (`:1634`), reset checkout data / clear coupons (`:1637-1638`), activity log (`:1639-1641`).
  6. Publish `OrderPlacedEvent` (`:1644`); re-check status; if paid → `ProcessOrderPaidAsync` (raises `OrderPaidEvent` `:1112`).
- **Side effects** — **FACT**: DB writes (order, order items, usage history), payment provider call (external), customer activity log row, domain events (`OrderPlacedEvent`, `OrderPaidEvent`, `OrderStatusChangedEvent` `:1025`) that plugins/consumers can handle, notification emails queued.
- **Failure modes** — **FACT**:
  - All work is wrapped in `try/catch`; exceptions are logged via `_logger.ErrorAsync` and converted into `PlaceOrderResult` errors (`:1661-1676`) — **failures are swallowed into a result object, not thrown to the caller**. **INFERENCE**: a partial failure after `SaveOrderDetailsAsync` but before later steps could leave side-effects half-applied; there is no explicit DB transaction visible around the whole `placeOrder` body (**Tentative** — verify by inspecting `SaveOrderDetailsAsync` and whether an ambient transaction/unit-of-work exists).
  - Optional pessimistic locking: when `_orderSettings.PlaceOrderWithLock` is set, the order is placed under a distributed lock keyed by customer id (`:1679-1683+`) — **INFERENCE**: this guards against double-submit/duplicate orders, and is **off by default** unless configured.
- **Where errors are costly** — **INFERENCE**: payment capture vs. order persistence consistency; duplicate-order risk when locking is disabled; event handlers with side-effects that run after payment.

### Flow B — Inbound HTTP request pipeline
- **Trigger/entrypoint** — **FACT**: `app.ConfigureRequestPipeline()` (`Program.cs:47`) delegates to `EngineContext.Current.ConfigureRequestPipeline(application)` (`ApplicationBuilderExtensions.cs:52-55`), which invokes each `INopStartup.Configure` in `Order` sequence. **INFERENCE**: middleware order is therefore determined by plugin/startup `Order` properties, not a single linear file.
